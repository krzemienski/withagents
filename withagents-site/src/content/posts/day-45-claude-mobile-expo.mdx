```typescript
// This hangs forever in React Native.
const response = await fetch(url);
const reader = response.body.getReader();
const { value } = await reader.read(); // never resolves
```

That is the single line that forced `claude-mobile-expo` to exist as its own binary instead of collapsing back to a tunnel-config pattern like Day 44.

I spent two sessions convinced it was a polyfill problem. It was not. The React Native fetch implementation does not support `ReadableStream.getReader()`, and the Server-Sent Events stream that Claude Code uses to push agent output is exactly the case `getReader()` was designed for. The combination is a soft lock. No error. No timeout. Just a promise that never resolves.

The fallback that shipped is XMLHttpRequest with `onreadystatechange`:

```typescript
const xhr = new XMLHttpRequest();
xhr.onreadystatechange = () => {
  if (xhr.readyState === 3) { // LOADING — progressive chunks arrive here
    processResponseText(xhr.responseText);
  }
};
```

React Native delivers partial SSE frames through `readyState === 3` (LOADING), which is what the fetch stream API should have handled. The pattern is ugly. It works. `claude-mobile-expo` ships against a real Claude Code backend on every test gate.

## Why this product exists at all

Yesterday's post argued that SDK-mode products (opencode) get a tunnel and an auth policy, not a native client. This post is the argument for why Interactive-mode products do earn a native client.

Claude Code on mobile is not just a shell-over-tunnel. It is a skill library, an MCP config, a session state you want to resume across devices, a set of hooks firing on every tool call. The surface area is the product. Punting to a web wrapper loses the parts of the product that matter most: offline skill browsing, TestFlight-grade push notifications when an agent stalls, a first-class file picker for attaching context.

So the v2.0 binary ships with:

- React Native frontend against a Python FastAPI backend providing an OpenAI-compatible API for the Claude Code CLI.
- 22/22 frontend tests, 40/40 backend tests, 22/22 integration tests passing against the real CLI (not a mock).
- Five gate checkpoints (P1, F1, F2, I1, I2) all required to pass before a binary was cut.
- Flat-black theme tuned to match the Hyper-Black + Ultraviolet brand palette, so the mobile surface reads as part of the same system as withagents.dev.

The tests are not unit tests. They are gate tests running against a booted backend and a real Claude CLI subprocess. When the SSE stream fails, the gate fails, and nothing ships. That discipline is what pulled the `fetch().getReader()` bug out in the first place. A unit-test mock would have reported green.

## What the 84/99 figure actually means

The production-ready sticker on the README says "84/99 tests (84.8%) — All gates passed." Both halves of that sentence are honest and need reading together.

- The 84 passing tests are the ones that gate the binary. If any of them fail, the build does not go to TestFlight. That is the half that gets read as "production ready."
- The 15 non-passing tests are load-shape tests that require infrastructure I have not finished wiring (multi-device push, background audio, iCloud-backed session sync). None of them block a normal user.

The distinction matters for a brand that will not say "production ready" without the real artifact behind it. 84/99 with hard gate coverage beats 99/99 on a mock harness every time. The tests that gate real agent streams caught the XHR bug. Tests that mock the stream would have reported the code correct.

## What went into the SSE fix, timed

- ~45 minutes reading React Native issues #27741 and #25910, both of which document the fetch-streams gap clearly and both of which I had skimmed before and dismissed.
- ~20 minutes trying a `react-native-sse` polyfill. The polyfill works for pure event-stream, does not work for the mixed text/event-stream + tool-call JSON frames Claude Code emits.
- ~30 minutes writing the XHR client in `src/services/http/sse.client.ts` with the `readyState === 3` chunk handler and a buffer that flushes on `\n\n`.
- ~15 minutes running the integration gate I2 against a real CLI session, watching the chunks tick in the simulator console, and marking the gate green.

Total: about two hours of focused work after the correct diagnosis. Five sessions of the wrong diagnosis before that. The ratio of "wrong diagnosis time" to "fix time" is the one I keep needing to show myself.

## The part that is still gross

`readyState === 3` gives you the whole response body buffered to this point. You re-parse the buffer every time. For a 45-minute agent session with multiple tool calls, you are re-tokenizing a multi-megabyte string on every chunk. The fix in place is a checkpoint offset: keep the last-parsed index, slice from there forward. I have not load-tested it against the worst real session I have on disk (the 657MB sessionforge arc).

If that breaks at scale, the answer is a proper TurboModule in native code that exposes a stream. I have avoided that because it is a platform-specific maintenance burden on a repo that is otherwise cross-platform React Native. I do not have a clean answer yet.

## What the mobile story teaches about mode-bets

Three products, three mobile stories:

1. **opencode (SDK):** tunnel config, four lines of JSON. Day 44.
2. **claude-mobile-expo (Interactive):** native RN client, real backend, real SSE against a real CLI. This post.
3. **ralph-orchestrator (Non-Interactive):** status and kill-switch surface. A different post.

The mode-bet tells you what kind of mobile product you are building. It does not tell you the SSE stream is going to hang forever the first time you try to read it. That part is on React Native.

Ship against the real wire. Fail on the real wire. That is the whole series.
