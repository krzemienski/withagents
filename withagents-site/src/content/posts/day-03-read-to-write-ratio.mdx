88,560 Reads. 10,140 Writes. 360 days.

That is the full session mine across 72 projects, 2.76 million lines of JSONL, 8.15 GB on disk. One number in the tool leaderboard explains every orchestration decision in this brand: **9.6:1 Read-to-Write ratio**. Agents spend almost ten turns understanding for every turn producing.

The demo loop hides this. A product demo surfaces the Write. The user sees the file appear, the commit land, the PR open. The Read is invisible: file opens, directory walks, search queries, grep scans, symbol lookups. The Write is one line of the session JSONL. The Read is most of the session.

## The tool leaderboard

From the 360-day mine across 72 projects (full data at `scripts/output/mine-360d-data.json`):

| Tool | Invocations | Mode |
|------|-------------|------|
| Read | 88,560 | Understanding |
| Bash | 75,658 | Execution + inspection |
| Grep | 22,463 | Understanding |
| Glob | 11,773 | Understanding |
| Write | 10,140 | Production |
| Edit | 19,979 | Production |

Add the understanding tools: Read + Grep + Glob = 122,796 invocations. Add the production tools: Write + Edit = 30,119. The understanding side outweighs production 4.1 to 1 even after Edit is counted as output. Restrict the comparison to Read vs Write as the two atomic primitives and the ratio is 9.6:1.

## What that ratio funds

Every orchestration decision I made across those 360 days flows from this. Agents are not "code generators with memory." They are readers that occasionally write, and the Write is the last 10% of the job.

Three design consequences:

**1. Context curation is the bottleneck, not generation capacity.** A 200K-token context window burned on irrelevant files is worse than a 40K window burned on the five files that matter. Hat-scoped sessions (see Day 07) convert the same budget into 94% task completion at 2% contradiction rate; monolithic sessions burn the same tokens and land at 67% with 34% contradiction. The Read budget is the agent's real working memory.

**2. File system over shared state.** Agents cannot share memory. They can share files. When ninety percent of the work is reading, the only coordination primitive that scales past two agents is the filesystem itself. Plans written to disk, task lists as files, build results as JSON events. Four thousand two hundred thirty-seven TaskUpdate calls across the mine all coordinate through disk artifacts, not runtime state.

**3. Evidence before output.** A session that reads five files and writes one line is doing the expected shape of the work. A session that writes ten files after reading one is flying blind. I built the `completion-claim-validator.js` hook because the tool sequence reveals the truth the reasoning prose hides. An agent can narrate confidence. The Read-to-Edit ratio cannot lie.

## The inverse pattern is the tell

Sessions where Edit outpaces Read correlate tightly with rework in the next session. I checked this against the observation store: sessions with Read-to-Edit below 2:1 produce edits that get reverted or amended in the following session roughly three times as often as sessions that sit above 4:1. The agent that writes without reading is the agent whose work gets rolled back.

This is not a heuristic the agent surfaces in its reasoning trace. It surfaces in the tool sequence. Forty-seven Reads before one Edit is an exploration session. Twelve Edits with zero preceding Reads is blind generation. Both are visible in the raw log. Neither requires a language model to detect.

## What this means for product design

Agent products that optimize for Write volume optimize for the wrong axis. The axis that matters is the quality of the Read that preceded the Write, measured by whether the Write survives the next session.

I say this as someone who shipped a year of Claude Code skills and watched the skills that helped most were the ones that constrained the Read phase: `/oh-my-claudecode:deep-dive`, `/start:analyze`, structured scouting with parallel agents. The skills I used least were the ones that tried to generate faster. Speed was never the problem. Grounding was the problem.

The **Memory Layer** and **Runbooks** product pillars on this site are both built against this. Memory Layer is about preserving the Read state across sessions so the next agent does not restart the understanding phase from zero. Runbooks are about encoding the Read patterns that actually worked, so the agent reads the right files in the right order, not a breadth-first walk of the repo.

If you are designing an agent system, the question worth asking is not "how fast can it generate?" It is: **what is the shape of the Read that has to happen before this Write is safe?**
