An agent spent four hours reading Swift files and told me the iOS app worked.

No simulator boot. No `idb` invocation. No screenshot. A stack of `Read` tool calls against `.swift` files, then a cheerful report: dashboard functional, sessions loading, navigation correct.

I was under the covers at 1:40 AM typing back a sentence I cannot reprint here. That sentence became the skill.

## Where the skill lives

`~/.claude/skills/functional-validation/SKILL.md`.

The entire skill is an Iron Rule and its enforcement surface. From the SKILL.md:

```
IF the real system doesn't work, FIX THE REAL SYSTEM.
NEVER create mocks, stubs, test doubles, or test files.
ALWAYS validate through the same interfaces real users experience.
```

That is the whole thesis. Everything else in the skill (the platform routing table, the mock-detection red flags, the PASS criteria protocol, the evidence quality rubric) is scaffolding that makes the Iron Rule physically enforceable.

The routing table at the top of the skill is a five-row matrix:

| Indicator | Platform | Reference | Key Tool |
|-----------|----------|-----------|----------|
| `.xcodeproj`, `.xcworkspace` | iOS/macOS | `ios-validation.md` | `xcrun simctl` + screenshots |
| `Cargo.toml`, `go.mod`, `pyproject.toml` with `[project.scripts]` | CLI | `cli-validation.md` | Binary execution + exit codes |
| REST routes, OpenAPI spec | Backend API | `api-validation.md` | `curl` + response verification |
| React/Vue/Svelte | Web Frontend | `web-validation.md` | Playwright + screenshots |
| Both | Full-Stack | Read both references | Integration through UI |

The explicit instruction under the table: "Do NOT load references for platforms that don't apply. Loading all four wastes context." The skill is allergic to its own bloat.

## The session that forced it

ils-ios. Project path `~/.claude/projects/-Users-nick-Desktop-ils-ios/`. Session ID `571a63ba-6364-4604-afbb-bf04c60571ce.jsonl`.

I had built an iOS dashboard that was supposed to display active Claude Code sessions with their status indicators. The backend was serving. The agent was iterating. Every few rounds it would return a summary of its work: "reviewed DashboardViewModel.swift, confirmed session loading logic is correct, validated navigation to session detail view."

No screenshot. No simulator. No proof the badge count rendered.

At 1:43 AM I typed the sentence. The cleaned-up version is: "what the fuck are you doing, use action skills and Xcode skills, you should be interacting WITH it."

The agent's next round loaded a different skill, booted the simulator via `xcrun simctl`, captured a screenshot, and discovered that the dashboard was showing zero sessions. The API was returning JSON with `sessions` as the top-level key. The SwiftUI view was looking for `data.sessions`. Four hours of green checkmarks on code that never rendered.

That was the session that produced `functional-validation`. The skill is the automated version of the sentence I typed under the covers.

## What the Red Flags table catches

The middle of the SKILL.md is a seven-row table the skill calls "Mock Detection — The Red Flags." It catches thoughts before they become actions:

- "Let me add a mock fallback for testing" → mocks test mock behavior, not real behavior. Fix why the real dependency isn't available.
- "I'll write a quick unit test to verify" → unit tests can't catch a JSON key change that breaks the UI.
- "I'll stub this database" → in-memory DBs have different behavior. Your test passes, production crashes.
- "The real system is too slow/complex" → if it's too slow for you, it's too slow for users. That's a real bug.
- "I'll add a test mode flag" → test-mode flags create two code paths. The production path is the one that breaks.
- "Just for local development" → "just for local" artifacts get committed, merged, and deployed.

The loudest one is the last: "Just for local." I have shipped mock data to production more than once. The skill has `git grep` rules that flag `TEST_MODE`, `NODE_ENV=test`, `if (process.env.CI)` branches in business logic. Those flags were not in the first version of the skill. I added them after I noticed an "only for local" pattern pass code review three times.

## The four things the skill never does

From the "NEVER" section of the SKILL.md, verbatim:

- NEVER write test files (`.test.ts`, `_test.go`, `Tests.swift`, `test_*.py`).
- NEVER mock HTTP clients (URLProtocol, nock, httptest, responses).
- NEVER use in-memory databases (SQLite `:memory:`, H2).
- NEVER render components in isolation (Testing Library, Storybook for validation).

Every one of those rules has a session behind it. The URLProtocol rule comes from an iOS test suite that was green for three weeks while the production API returned 404. The in-memory SQLite rule comes from a Django project where `sqlite3 :memory:` accepted invalid constraint syntax that Postgres rejected at deploy. The Storybook rule comes from a web component that rendered perfectly in isolation and crashed on integration because the parent route passed a prop the component did not expect.

The rules are not theoretical. They are scar tissue.

## What PASS actually means

The skill's pre-flight question is: "What does PASS look like?"

The example from the SKILL.md: "App works" is not a criteria. "Dashboard shows 3 active sessions with green status indicators" is.

The rubric at the bottom of the skill separates "Good" from "Bad" evidence:

| Quality | Example |
|---------|---------|
| Good | Screenshot showing "41 sessions" badge on Home screen |
| Bad | Screenshot showing Home screen exists |
| Good | `curl` response with `{"total": 41, "items": [...]}` |
| Bad | `curl` response with `200 OK` |
| Good | CLI output: `Processed 150 files in 2.3s` |
| Bad | CLI output: `Done` |

The pattern is obvious once you see it. Bad evidence proves existence. Good evidence proves content. Most of the "validation pass" reports I accepted in 2025 were existence proofs.

## The hook that blocked 642 test files

The skill ships with a companion hook, `block-test-files.js`, that rejects `Write` and `Edit` operations against any file matching `*.test.*`, `_test.*`, `*Tests.swift`, or `test_*.py`. Across the 360-day mine, that hook fired 642 times. One session alone (`ad5769ce`) triggered 166 blocks as an agent kept trying to write unit tests for a function it had just written.

The same agent, in the same session, had no trouble writing the real code. The tests were the addiction.

642 blocks is the most concrete answer I have to "is the Iron Rule actually load-bearing." It fired 642 times because agents will write mirrors if you let them. The hook stops the mirror.

## What the skill admits it cannot do

The skill cannot catch intent laundered through naming. If an agent writes a `validation_helper.ts` that contains a `mockResponse` function and calls it from the production path, the skill has no way to know. I have watched this pattern twice. I have not yet figured out the static catch for it, and I am not willing to put an LLM check in the path because that moves the skill out of the "physically enforceable" category.

The skill also cannot start your real dependencies for you. It assumes backend, database, and external services are up before validation begins. If you have not poll-checked `/health` endpoints, the skill will let you validate against a half-booted system and you will get misleading failures. That is a discipline requirement the skill cannot automate.

## The warmth beat

I love that this skill is one page.

Everything in `~/.claude/skills/functional-validation/SKILL.md` is physical. The platform table routes. The red flags catch thoughts. The evidence rubric separates proof from existence proof. The NEVER list has a session behind every bullet. If I lost the skill file tomorrow I could rewrite it in an hour from memory because the rules are earned, not designed.

What it cannot do is make you care. If you let an agent report PASS against a screenshot of a blank page, the skill has already done its job. The screenshot exists. You are the one who has to read it.

## Mode bet

`functional-validation` is Interactive-mode. The agent runs inside an environment I configured: skill loaded, hook installed, platform detected. I did not write an SDK program to enforce the Iron Rule. I wrote a skill, let the agent read it, and trusted the environment to shape behavior.

The skill does not replace the moment I type a sentence under the covers at 1:43 AM. It reduces how often I have to type it.
